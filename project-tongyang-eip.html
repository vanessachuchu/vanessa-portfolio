🎯 專案背景：跨國企業的系統整合挑戰

東陽實業的困境

東陽實業作為跨國汽車零配件集團，業務遍布北美、南美、歐洲多國，擁有數千名員工和39 個不同職級與部門角色。然而，企業內部卻面臨著嚴重的系統分散化問題。

企業面臨的三大痛點

痛點 1：系統分散，缺乏統一入口
  • • 員工需要使用多個獨立系統：Notes、智庫、訊息平台、業績系統等
  • • 每個系統都有獨立的帳號密碼，記憶負擔大
  • • 系統間資訊無法整合，需要重複查詢
  • • 新進員工需要學習多個系統的操作方式
痛點 2：國際化營運的複雜性
  • • 北美、南美、歐洲多個地區，員工使用不同語言
  • • 各地區有不同的系統需求和功能權限
  • • 缺乏統一的多語系支援機制
  • • 地區化配置管理困難
痛點 3：權限管理複雜且容易出錯
  • • 39 個不同角色，涵蓋高階主管到基層員工
  • • 每個角色對不同系統有不同的存取權限
  • • 手動管理權限容易遺漏或設定錯誤
  • • 權限變更缺乏完整的追蹤機制

專案的核心目標

建立一個統一的企業入口網站（EIP），整合所有既有系統，實現單一登入（SSO）、精細化權限管理、多語系支援，提升員工工作效率並強化資訊安全管控。

💡 解決方案：系統化的整合設計方法

第一步：統一身份認證與單一登入

挑戰：如何讓員工只登入一次就能使用所有系統？

東陽實業內部有多個獨立開發的系統，每個系統都有自己的認證機制。要實現真正的單一登入，需要整合這些異質系統的認證方式。

設計決策 #1：AD 域整合 + 多重登入機制

我設計了三層式的認證架構：

  • 第一層：OS 層自動代登入 - 當員工在公司電腦登入 Windows AD 帳號後，開啟 EIP 系統會自動完成認證，無需再次輸入帳號密碼
  • 第二層：手動登入機制 - 提供標準的帳號密碼登入方式，支援非 AD 環境的使用者
  • 第三層：系統間代登入 - EIP 作為中介平台，透過 TYGOTP 和 WebTicket 機制，實現對其他系統的自動登入

技術亮點：角色登入 vs. 使用者帳號登入

在設計系統連結機制時，我發現了一個關鍵問題：不同系統有不同的登入方式需求。

🔍 登入方式設計的思考過程

情境 1：角色登入系統

某些系統需要根據使用者的「角色」來決定登入後看到的內容和權限。例如：主管角色和員工角色登入同一個系統後，看到的功能選單完全不同。

情境 2：使用者帳號登入系統

另一些系統則是根據「個人帳號」來識別使用者，不需要選擇角色。例如：個人郵件系統、個人行事曆等。

我發現的設計盲點：
  • • 如果系統從「角色登入」改為「使用者帳號登入」，原先已綁定該系統的使用者會無法登入
  • • 如果系統從「使用者帳號登入」改為「角色登入」，但沒有設定任何角色，系統連結會變成無效狀態
  • • 如果某個「角色登入」系統的所有角色都被刪除了，已經加入系統連結的使用者該怎麼辦？
設計決策 #2：登入方式切換的保護機制

我設計了一套完整的保護邏輯：

  • 規則 1：已經綁定角色的系統，不可將最後一組角色刪除（避免系統連結失效）
  • 規則 2：系統選單不可顯示尚未綁定角色的「角色登入」系統（避免使用者選擇後無法進入）
  • 規則 3：登入方式變更時，系統會即時檢查並提示可能受影響的使用者數量
  • 規則 4：所有登入方式的變更都記錄在系統日誌中，可追溯查詢

業主的反應：業主原本沒有想到這些邊緣情境，在我提出這些問題並展示解決方案後，非常認同這套保護機制的必要性，避免了未來可能發生的系統混亂。

第二步：39 個角色的權限管理設計

挑戰：如何設計一套既精細又易於管理的權限系統？

東陽實業有 39 個不同的使用者角色，從董事長、總經理、各部門主管到基層員工，每個角色對系統的存取需求都不同。

📋 權限設計的核心思維

思考點 1：權限的顆粒度

權限應該細到什麼程度？如果太細，管理成本很高；如果太粗，無法滿足精細化需求。我採用的策略是：系統層級 + 功能模組層級的兩層式權限控制。

思考點 2：權限的繼承與覆蓋

某些角色之間有階層關係，例如部門主管應該擁有部門員工的所有權限，再加上管理權限。我設計了權限繼承機制，避免重複設定。

思考點 3：權限變更的即時性

當角色權限被修改時，已登入的使用者是否需要重新登入？我設計了權限快取機制，並提供「強制重新載入」選項。

實作策略：角色-系統對應表

我建立了一個完整的「角色-系統權限對應表」，清楚記錄每個角色可以存取哪些系統和功能。這個對應表不僅是技術文件，更成為業主管理權限的重要參考依據。

設計亮點：批量權限操作
考慮到實際管理需求，我設計了批量權限操作功能：可以一次性為多個角色設定相同的系統權限，或是一次性為某個角色設定多個系統的權限。這大幅降低了管理者的操作負擔。

第三步：多語系與國際化支援

挑戰：如何支援全球不同地區的使用需求？

東陽實業的員工分布在不同國家，使用不同語言。系統需要支援繁體中文、英文、印尼文、越南文。

🌍 國際化設計的考量點

語系切換邏輯
  • • 自動偵測：系統會根據瀏覽器的語系設定自動顯示對應語言
  • • 手動切換：使用者可以隨時切換語系，選擇會儲存在個人偏好設定中
  • • 多語系內容管理：後台提供獨立的多語系內容編輯介面
地區化配置
  • • 不同地區可以有不同的系統連結顯示
  • • 時區、日期格式、數字格式的地區化處理
  • • 地區專屬的公告訊息管理

第四步：15 個功能模組的整合設計

EIP 系統整合了 15 個核心功能模組，每個模組都需要仔細設計使用者體驗和系統整合方式。

模組設計策略：統一框架 + 獨立內容

我設計了一個統一的版面框架，所有模組都遵循相同的視覺風格和互動邏輯。但每個模組的內容來源不同：

  • • 原生開發模組：訊息公告、我的最愛、系統地圖等，完全由 EIP 系統自行管理
  • • Iframe 嵌入模組：訊息平台、AM 行銷物流公告等，透過 iframe 嵌入既有系統
  • • API 整合模組：個人郵件、行事曆、待辦事項等，透過 Notes AP 的 API 整合
  • • 混合模式模組：智庫搜尋、產業新聞等，結合 API 和頁面跳轉

📊 專案執行：9 個月的開發歷程

第一階段：需求分析（3-4月，73天）

這個階段的重點是深度理解企業需求和盤點既有系統。

需求盤點的關鍵工作

• 訪談各部門主管和關鍵使用者，了解實際工作流程
• 盤點所有需要整合的既有系統，評估整合難度
• 整理 39 個使用者角色的定義和權限需求
• 確認 AD 域認證的技術細節和整合方式

第二階段：系統開發（4-8月，243天）

這是整個專案最長的階段，包含基礎建置、前台開發、系統整合三大部分。

開發策略：漸進式整合
我採用漸進式整合的開發策略，不是等所有功能都開發完才開始整合，而是每完成一個模組就立即整合測試。這樣可以更早發現整合問題，避免最後階段出現大量整合問題。

第三階段：整合測試（8-11月，83天）

63 天的整合測試和 20 天的 UAT 驗收測試，確保系統品質。

測試重點項目

功能完整性測試
  • • 15 個功能模組的完整測試
  • • 39 個角色的權限驗證
  • • 多語系切換測試
跨系統整合測試
  • • AD 域登入驗證
  • • Notes AP 整合測試
  • • Iframe 嵌入系統測試

第四階段：交付文檔（11-12月，31天）

我花了 20 天撰寫32 頁完整的前台操作手冊，涵蓋所有功能的詳細操作說明。

🎓 核心學習與反思

1. 企業級專案的複雜度遠超想像

39 個角色、15 個功能模組、多個既有系統整合，每個細節都需要仔細思考。專案經理需要有能力將複雜問題拆解成可管理的小問題，逐一解決。

2. 主動發現問題比被動回應問題更有價值

在登入方式切換、權限管理等議題上，主動思考可能的邊緣情境和問題，提前設計保護機制，避免了未來可能發生的系統混亂。

3. 使用者體驗與安全性需要平衡

單一登入提升了便利性，但也需要考慮安全性。需要在兩者之間找到平衡點，既方便使用又不損害安全。

4. 完整的文件是專案成功的關鍵

32 頁的操作手冊不僅是交付物，更是使用者培訓、系統維護、未來擴充的重要依據。投入時間撰寫完整文件，長期來看是值得的。
