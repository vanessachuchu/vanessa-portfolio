專案起源：複雜度超乎想像的挑戰

YesOnline 不是一個簡單的線上學習平台。當我接手這個專案時，面對的是一個需要整合學生、顧問、教師三個核心角色的複雜生態系統，還要串接第三方教學平台 ClassIn 的 API，支援兩個不同機構的多租戶架構。

更大的挑戰在於：設計師交付的 Figma 檔案只有視覺呈現，沒有任何業務邏輯說明。每個按鈕點擊後會發生什麼？資料如何驗證？異常狀況如何處理？這些都需要產品經理自行推敲、設計、並文件化。

💡 產品經理的核心挑戰

視覺與邏輯的鴻溝：Figma 只有畫面，業務流程需自行設計
跨團隊溝通：向上確認需求，向下釐清邏輯，橫向協調設計
業務複雜度：堂數管理、時段排程、衝突檢測、多機構隔離
品質保證：20+ 測試腳本，涵蓋每個操作路徑

從零開始：系統架構的思考與設計

在進入開發前，我花了三週時間進行系統分析與架構設計。這不僅是技術架構的規劃，更是對整個業務模式的深度理解。

多租戶架構：機構資料的完全隔離

YesOnline 需要同時服務 Yesonline 和 HsinFei 兩個機構，每個機構都有各自的教師、學生、課程資料。這看似簡單的需求，背後隱藏著複雜的資料隔離設計。

設計決策 #1：課程 ID 編碼規則
我設計了一套 17 碼的課程 ID 編碼系統，包含：機構簡稱（3碼）+ 課程模式（1碼）+ 建立時間（8碼）+ 流水號（5碼）。這不僅解決了資料隔離問題，更讓每個課程都能追溯其完整資訊，為後續的資料分析奠定基礎。

三角色權限體系：從業務邏輯到技術實現

學生、顧問、教師三個角色，各自有不同的功能權限和資料存取範圍。在設計階段，我就必須思考清楚：

🎯 角色權限設計思考

學生端：只能看到自己的課程、堂數、行事曆，選課時只能看到有空缺的課程
顧問端：可以管理所有課程、教師時數、學生堂數，但不能看到其他機構的資料
教師端：只能查看自己被排定的課程，無法存取其他教師的資料

業務邏輯設計：補足設計稿的空白

產品經理最大的價值，在於能夠從視覺設計中推敲出完整的業務邏輯。以學生選課流程為例，Figma 只有五個畫面，但我需要思考的是：

選課流程：六個步驟背後的細節設計

📋 選課流程的業務邏輯設計

步驟 1：選擇課程種類
• 學生只能看到已購買的課程種類
• 需即時顯示剩餘可選課堂數
• 若剩餘0堂，該課程種類應如何呈現？（設計稿沒說明）
步驟 2：選擇時間區間
• 開始日期與結束日期可以是同一天嗎？（可以）
• 時間區間的最小單位是什麼？（30分鐘）
• 結束時間必須晚於開始時間，如何驗證？
步驟 3：選擇教師與時段
• 教師列表需依照什麼規則排序？（最近更新）
• 若某教師該時段所有課程都滿了，是否顯示？（不顯示）
• 若某教師部分時段已滿，如何標示？（已滿時段不顯示在行事曆）
• 學生一次可以選擇幾堂課？（依剩餘堂數）
步驟 4：時段衝突檢測
• 學生不能選擇自己已有課程的時段（即時檢測）
• 若選擇重複時段，如何提示？（彈窗警示）
• 已選課的時段在行事曆上如何標示？（disabled 狀態）
步驟 5：確認資訊
• 顯示所有已選課程明細，確保學生清楚知道選了什麼
• 提供取消單筆選課的功能，但不影響其他已選課程
• 確定後才扣除堂數，避免中途取消造成堂數異常
步驟 6：選課成功
• 顯示成功訊息，5秒後自動跳轉
• 提供「繼續選課」按鈕，方便學生一次選多堂
• 更新學生端課程列表與行事曆
與業主的溝通實例
在選課時段衝突的處理上，我主動詢問業主：「如果學生選擇的時段與已有課程重疊，系統應該即時阻擋，還是在最後確認時才提示？」業主原本沒想過這個問題，經過討論後決定採用「即時檢測」方案，避免學生選到最後才發現無法提交的挫折感。這種主動發現盲點的能力，正是產品經理的核心價值。

堂數管理：複雜業務邏輯的系統化設計

堂數管理是整個平台最複雜的模組之一。顧問需要為學生建立堂數訂單，每個訂單包含課程種類、課程方案（一般/吃到飽）、購買堂數、使用效期等資訊。看似簡單的 CRUD 操作，背後需要考慮的邏輯卻異常複雜。

訂單編號的唯一性驗證

當顧問新增堂數時，我發現設計稿沒有說明訂單編號的規則。這引發了一連串的思考：

🔍 訂單編號設計的思考過程

• 訂單編號是顧問手動輸入，還是系統自動產生？（與業主確認：手動輸入）
• 如果手動輸入，如何確保不會重複？（新增前需檢查唯一性）
• 重複時應該如何提示？（紅字錯誤訊息，阻擋提交）
• Excel 匯入時，若有重複編號應如何處理？（匯入失敗，顯示錯誤明細）

堂數回補機制：業主沒想到的細節

在堂數管理中，有一個「回補堂數」功能。當學生因故無法上課時，顧問可以將該堂課的堂數回補給學生。這個功能在設計稿中只有一個「回補」按鈕，但我需要設計完整的業務邏輯：

設計決策 #2：回補堂數的完整流程
我設計了一套「回補-收回」的雙向機制。回補後，若發現誤操作，顧問可以執行「收回」動作。但這引發了新問題：如果學生已經用回補的堂數選了新課程，還能收回嗎？經過與業主討論，我們決定：收回時需先檢查學生是否已使用該堂數，若已使用則不允許收回，避免資料異常。這種細節設計，都需要產品經理主動思考並提出解決方案。

跨團隊溝通：文件化的重要性

當業務邏輯越來越複雜，口頭溝通已經無法確保資訊的準確傳遞。我建立了一套完整的文件體系，包含：

📚 文件產出與溝通策略

給業主的文件

  • ✅ 功能規格說明書（每個頁面的完整流程）
  • ✅ 業務邏輯確認文件（需決策的細節）
  • ✅ 異常情境處理方案
  • ✅ Excel 匯入範本與規則

給工程師的文件

  • ✅ 欄位驗證規則（必填/選填/格式/長度）
  • ✅ API 規格與參數說明
  • ✅ 資料庫欄位設計
  • ✅ 錯誤訊息文字內容

工作功能說明書：細節的極致呈現

我撰寫的工作功能說明書，針對每個頁面都包含：頁面名稱、功能說明、欄位列表、欄位說明、預設值、驗證規則、彈窗內容、按鈕行為、異常處理。以「新建堂數」頁面為例：

📄 功能說明書範例：新建堂數

欄位名稱 資料型態 驗證規則 說明
訂單編號 文字 必填，唯一性檢查 送出前需檢查是否重複
學生 下拉選單 必填 姓名模糊查詢，僅顯示本機構學生
課程種類 下拉選單 必填 中英文並列，如：生活英語 General English
課程方案 下拉選單 必填 一般方案 / 吃到飽
使用效期 日期 必填，需晚於訂單時間 格式：YYYY-MM-DD

測試策略：品質保證的完整思維

為了確保產品品質，我規劃了20+ 測試腳本，涵蓋80+ 測試情境。每個測試腳本都遵循統一的結構：前置條件、測試步驟、預期結果、異常情境、跨平台驗證。

測試腳本的系統化設計

🧪 測試架構：五大核心模組

模組一：使用者管理
註冊流程、登入驗證、忘記密碼、個人資料編輯、密碼規則檢查
模組二：課程管理
單次課程建立、頻次課程建立、課程編輯、課程刪除/啟用、列表篩選
模組三：堂數管理
堂數新增、編輯、查看、回補機制、Excel 匯入驗證
模組四：時數管理
教師時數追蹤、列表查詢、詳情查看、重複防護
模組五：顧問端視圖
教師日曆、課程詳情、權限管理、資料隔離驗證
測試腳本實例：學生選課時段衝突檢測
前置條件：學生已登入，已有一堂 2023-05-10 14:00-15:00 的課程
測試步驟：選擇同一天 14:30-15:30 的課程
預期結果：系統即時顯示「此時段與已選課程重疊」錯誤訊息，無法勾選該時段
異常情境：若學生在另一台裝置同時選課，後端需再次驗證時段衝突
跨平台驗證：在 1440px 桌面版和 320px 手機版分別測試，確保錯誤訊息正確顯示

資訊安全：合規性的系統化設計

除了功能開發，資訊安全也是產品經理必須關注的重點。我在系統設計階段就規劃了完整的資訊安全架構：

🔒 資訊安全設計要點

密碼安全規範：
• 長度至少 8 位
• 需包含大寫、小寫、數字、特殊符號四類中的三類
• 不得重複使用前一次密碼
• 密碼需加密儲存，不得明碼
帳號鎖定機制：
• 登入失敗 3 次後鎖定帳號
• 至少 1 小時後才能重試
• 或由管理員重置後解鎖
Token 管理：
• Access Token 有效期 1 小時
• Refresh Token 有效期 96 小時
• 過期後自動登出
稽核軌跡：
• 記錄登入時間、IP、錯誤訊息
• 身分驗證失敗事件追蹤
• 資料操作軌跡記錄

產品經理的核心能力展現

YesOnline 專案完整展現了產品經理在複雜專案中的核心能力：

能力一：業務邏輯設計能力
從 Figma 視覺稿中推敲出完整的業務流程，補足設計師沒有說明的邏輯細節，將抽象的需求轉化為可執行的規格。
能力二：跨團隊溝通能力
向上與業主確認需求細節，主動發現業主沒想到的問題；向下為工程師釐清開發邏輯，提供完整的文件支援；橫向與設計師討論互動細節，確保使用者體驗的一致性。
能力三：系統性思維能力
從整體架構到細節實現，從正常流程到異常處理，從單一功能到模組整合，建立起完整的產品思維框架。
能力四：文件化能力
將所有業務邏輯、驗證規則、異常處理都轉化為清晰的文件，確保團隊成員對需求的理解一致，降低溝通成本。

結語：從零到一的產品思維

YesOnline 專案的價值不僅在於交付了一個功能完整的平台，更在於展現了產品經理如何從零開始建構一個複雜系統的完整過程。當面對只有視覺設計的 Figma 檔案時，產品經理的責任就是填補所有業務邏輯的空白，確保每個細節都經過深思熟慮。

214 個工作項目背後，是對每個功能模組的深度思考、對每個使用場景的完整規劃、對每個異常情境的妥善處理。這不是單純的需求整理，而是將業務需求轉化為技術規格、將使用者期待轉化為產品體驗的完整實踐。

這就是產品經理的核心價值：讓複雜變得清晰，讓抽象變得具體，讓想像變成現實。
