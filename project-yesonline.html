🎯 專案背景與挑戰

系統的複雜性

YesOnline 的複雜度體現在兩個維度：機構隔離與角色權限。

維度一：雙機構獨立架構

業務需求
系統需同時服務兩個獨立機構（Yesonline 和 HsinFei），每個機構擁有各自的教師資源、學生名單、課程設定，且資料必須完全隔離。

設計挑戰

  • • 如何確保機構 A 的顧問無法看到機構 B 的資料？
  • • 如何設計資料結構讓兩個機構共用同一套系統，又能保持資料獨立？
  • • 如何在未來擴充更多機構時，架構依然穩定？
我的解決方案
設計了一套完整的資料隔離機制，從資料庫結構、API 設計到前端權限控制，都納入機構識別邏輯。這不僅解決了當前需求，也為未來擴充更多機構建立了可擴展的架構基礎。

維度二：三角色權限體系

角色定義與權限設計

系統包含三個核心角色，各有不同的功能權限：

學生 (Student)
  • • 查看自己的課程、堂數餘額、上課行事曆
  • • 選擇課程、預約時段
  • • 僅能存取自己的學習資料
顧問 (Consultant)
  • • 管理所屬機構的所有課程設定
  • • 為學生建立和管理堂數訂單
  • • 追蹤教師授課時數
  • • 無法存取其他機構的資料
教師 (Teacher)
  • • 查看自己被排定的課程時間表
  • • 管理自己的課程內容
  • • 無法存取其他教師的資料或學生的個人資訊

設計思維
我需要思考的不只是「誰能看到什麼」，更重要的是「在什麼情境下能做什麼操作」。例如：學生選課時只能看到有空缺的課程；顧問編輯課程時需要即時檢查是否有學生已預約；教師查看課表時只顯示確認的課程，而非所有開放時段。

產品經理面臨的核心挑戰

挑戰 1：從視覺到邏輯的轉化

情境
設計師交付的 Figma 檔案只有畫面，沒有任何業務邏輯說明。

實際案例：學生選課功能

Figma 上只有五個畫面：選擇課程 → 選擇時間 → 選擇教師 → 確認資訊 → 完成。但我需要思考：

  • • 每個步驟的資料從哪裡來？
  • • 什麼情況下按鈕是 disabled 狀態？
  • • 如果學生選擇的時段已經被其他人選走了怎麼辦？
  • • 如果學生的堂數不夠了要如何提示？
  • • 選到一半離開頁面，已選的資料要保留嗎？

產品經理的價值
將這些設計稿中的「空白」轉化為完整的業務流程，確保每個細節都有明確的規則和處理方式。

挑戰 2：跨團隊溝通與決策

向上溝通
主動發現業主沒想到的問題，提供解決方案選項
向下溝通
提供完整的開發規格，讓工程師不需要猜測需求
橫向溝通
補足視覺設計中缺少的互動狀態

挑戰 3：複雜業務邏輯的梳理

每個功能模組背後都有複雜的業務邏輯：

  • 課程管理：單次課程 vs. 頻次課程，不同模式有不同的排程規則
  • 堂數系統：購買、使用、過期、回補，每個狀態的轉換都需要嚴謹的邏輯
  • 時段排程：需考慮教師可用時間、學生已選課程、機構時段規則
  • 衝突檢測：即時驗證時段是否重疊，避免重複預約

💡 解決方案：系統化的產品設計方法

第一步：建立系統架構思維

機構隔離的設計邏輯

設計原則
所有資料和操作都必須帶有「機構識別」，從系統最上層就確立這個請求屬於哪個機構，確保資料不會跨機構存取。

實施策略
  • • 設計唯一識別碼系統，讓每筆資料都能追溯所屬機構
  • • 建立權限檢查機制，在 API 層就阻擋跨機構存取
  • • 規劃資料庫結構，支援未來更多機構的擴充

為什麼重要？
這不只是技術架構問題，更是資訊安全和客戶信任的基礎。任何一個資料外洩的漏洞，都可能造成嚴重的商業損失。

第二步：業務邏輯的完整設計

以使用者旅程為核心的設計方法

實例：學生選課流程

設計師的 Figma 只有五個畫面，但我設計了完整的六步驟流程：

學生選課完整流程設計

步驟 1：選擇課程種類
  • • 學生只能看到已購買的課程種類
  • • 即時顯示每個課程種類的剩餘堂數
  • • 若剩餘 0 堂則無法選擇
步驟 2：設定時間範圍
  • • 設定想要選課的日期區間
  • • 系統驗證時間邏輯（結束時間需晚於開始時間）
步驟 3：選擇教師與時段
  • • 顯示該時段有開課的教師列表
  • • 以行事曆方式呈現教師的可預約時段
  • • 已滿額的時段自動隱藏或標示為不可選
步驟 4：即時衝突檢測
  • • 學生選擇時段時，系統即時檢查是否與已選課程重疊
  • • 若有衝突立即提示，無法勾選該時段
步驟 5：確認選課資訊
  • • 顯示所有已選課程的完整資訊
  • • 提供取消單筆選課的功能
  • • 確認後才扣除堂數，確保資料正確性
步驟 6：完成並更新
  • • 顯示成功訊息
  • • 自動更新學生的課程列表和行事曆
  • • 提供「繼續選課」選項

設計價值
這不只是畫面的排列，而是完整考慮使用者體驗、資料驗證、異常處理的系統化流程設計。

主動發現問題並提出解決方案

實例：堂數回補機制的設計

原始需求：當學生因故無法上課時，顧問可以回補堂數。

我發現的問題：

  • • 回補後，如果發現是誤操作怎麼辦？
  • • 如果回補的堂數已經被學生用來選新課程了，還能收回嗎？
  • • 回補和收回的操作需要記錄嗎？

我提出的解決方案：

設計「回補-收回」雙向機制：顧問可以回補堂數給學生；若誤操作，可以執行「收回」；收回前系統會檢查該堂數是否已被使用；若已使用則不允許收回，避免資料異常；所有操作都記錄操作者和時間，建立稽核軌跡。

業主的反應：業主原本沒想過這些細節，經過討論後認同這套機制的必要性，並納入正式規格。

這展現了產品經理的核心價值：主動發現盲點，提出解決方案，協助決策

第三步：文件化與溝通策略

為什麼需要完整的文件？

當業務邏輯變得複雜，口頭溝通已經無法確保資訊準確傳遞。我建立了完整的文件體系，讓所有團隊成員對需求理解一致。

文件產出體系

給業主的文件

  • ✓ 功能規格說明書（每個頁面的完整流程）
  • ✓ 業務邏輯確認文件（需決策的細節）
  • ✓ 異常情境處理方案

給工程師的文件

  • ✓ 欄位驗證規則（必填/選填/格式/長度）
  • ✓ API 規格與參數說明
  • ✓ 資料庫欄位設計
  • ✓ 錯誤訊息文字內容

工作功能說明書：細節的極致呈現

我撰寫的工作功能說明書，針對每個頁面都包含：頁面名稱、功能說明、欄位列表、欄位說明、預設值、驗證規則、彈窗內容、按鈕行為、異常處理。

為什麼重要？
清晰的文件讓工程師不需要猜測需求，大幅降低溝通成本和開發時的來回修改。

第四步：品質保證的系統化思維

測試策略的規劃

規劃了20+ 測試腳本，將整個系統拆解為五大模組進行測試：

五大核心測試模組

1. 使用者管理模組
註冊、登入、權限驗證
2. 課程管理模組
課程建立、編輯、機構隔離驗證
3. 堂數管理模組
訂單建立、堂數使用、回補機制
4. 時段排程模組
教師時間管理、學生選課、衝突檢測
5. 角色權限模組
三個角色的功能權限和資料存取驗證

測試腳本結構
每個測試都包含：前置條件、測試步驟、預期結果、異常情境、跨裝置驗證

資訊安全的前置規劃

在系統設計階段就規劃完整的資訊安全機制：

密碼安全規範
長度、複雜度要求、不得重複使用舊密碼、加密儲存
帳號安全機制
登入失敗鎖定機制、Token 管理和過期處理

🎓 核心學習與反思

從這個專案學到什麼？

1. 產品經理不只是需求傳達者，更是業務邏輯的設計者

當設計稿只有視覺時，產品經理需要有能力推敲出完整的業務流程，這需要對業務本質的深度理解和系統性思考。

2. 主動發現問題的價值遠大於被動執行

最有價值的產品經理，是能在業主、設計師、工程師都沒想到的地方，主動發現盲點並提出解決方案。

3. 文件化是大型專案成功的關鍵

越複雜的專案，越需要清晰的文件。文件不只是給別人看，更是強迫自己把邏輯想清楚的過程。

4. 權限設計是系統安全的基礎

機構隔離和角色權限的設計，不只是功能需求，更是資訊安全和客戶信任的基石。
