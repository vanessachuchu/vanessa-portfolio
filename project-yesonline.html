專案背景：複雜度的挑戰

YesOnline 不僅僅是一個線上學習平台，而是一個需要整合學生、顧問、教師三個核心角色的複雜生態系統。作為產品經理，我面臨的首要挑戰是如何在保持系統複雜性的同時，確保每個角色都能獲得流暢的使用體驗。

這個專案的複雜度體現在多個層面：業務邏輯的精密設計、多角色權限的安全管控、第三方系統的深度整合，以及跨平台的一致性體驗。每一個環節都需要產品經理具備系統性思維和精準的執行力。

💡 產品管理核心挑戰

業務複雜度：堂數管理、時段排程、取消規則
技術整合：ClassIn API、權限系統、響應式設計
用戶體驗：三角色流程、錯誤處理、國際化
質量保證：全覆蓋測試、邊界條件、安全驗證

深度業務邏輯：選課系統的精密設計

以學生選課流程為例，這個看似簡單的功能背後隱藏著複雜的業務邏輯。通過深入分析測試腳本，我發現這套系統設計體現了產品經理對業務場景的深度理解。

▲ 學生選課流程測試驗證：從課程種類選擇到選課成功的完整流程

選課流程的六個關鍵節點

每個步驟都包含了精心設計的驗證機制：課程種類驗證 → 時間區間設定 → 教師時段選擇 → 衝突檢測 → 確認流程 → 成功回饋。

成功要素 #1：風險控制設計
系統在每個節點都設置了安全檢查機制，包括堂數有效期驗證、時段衝突檢測、以及可使用堂數的即時計算。這種多層級的驗證體系確保了業務邏輯的安全性和用戶體驗的流暢性。

測試策略：完整性思維的體現

產品經理的價值不僅在於功能設計，更在於對品質的把控。YesOnline 專案中，我建立了20+ 測試腳本的完整測試體系，涵蓋用戶管理、課程管理、堂數管理、時數管理、顧問端視圖五大核心模塊，確保每個功能在各種場景下的穩定性。

系統化測試架構：五大模塊全覆蓋

👤 模塊一：用戶管理測試

註冊流程驗證

  • • 必填欄位驗證（國碼、手機號碼）
  • • 手機號碼格式檢驗（支援多國格式）
  • • 簡訊驗證碼流程（60秒重發機制）
  • • 帳號重複註冊防護
  • • 密碼規則合規性檢查

登入與安全功能

  • • 忘記密碼重設流程（手機驗證 + 密碼規則）
  • • 個人資料編輯驗證（頭像、姓名、手機）
  • • 修改密碼功能（舊密碼驗證 + 新密碼確認）
  • • Token 過期處理機制

📚 模塊二：課程管理測試

課程創建功能（雙模式）

  • • 單次課程創建：課程類型選擇、教師篩選、時段選擇、必填欄位驗證
  • • 頻次課程創建：重複週期設定、批量時段生成、衝突檢測
  • • 教師名稱/專長條件篩選功能
  • • 已選時段去重機制

課程維護與查詢

  • • 課程列表篩選（課程名稱、教師名稱、課程類型、狀態）
  • • 課程編輯功能（學生人數上限驗證、資料完整性檢查）
  • • 課程刪除與啟用流程（學生退選處理、時段釋放機制）
  • • 課程狀態管理（服務中/已刪除）

📊 模塊三：堂數管理測試

堂數查看與編輯

  • • 堂數詳情頁面完整性（訂單編號、學生資訊、訂單時間、金額、備註）
  • • 課程詳情顯示（課程類型、課程方案、堂數、使用效期）
  • • 堂數編輯功能（不可變更欄位鎖定、可編輯欄位驗證）
  • • 課程方案切換邏輯（一般 ↔ 無限制）
  • • 使用效期限制檢查

⏰ 模塊四：時數管理測試

顧問時數追蹤

  • • 時數新增功能（教師選擇、授課時數、缺席時數、月薪、薪資狀態）
  • • 重複時數防護（當月已存在教師自動隱藏）
  • • 時數詳情查看（教師資料、頭像、統計資訊）
  • • 必填欄位驗證與錯誤提示

時數列表與查詢

  • • 教師名稱篩選（存在/不存在教師）
  • • 年份/月份選擇（有資料/無資料月份）
  • • 預設顯示上月時數清單
  • • 分頁與排序（最新至最舊）

教師時數詳情

  • • 課程清單資訊（課程名稱、時間、類型、堂數、狀態、備註）
  • • 課程名稱篩選功能
  • • 日期/月份篩選
  • • 課程狀態分頁（已授課/未出席）

📅 模塊五：顧問端視圖管理測試

教師日曆查看

  • • 教師名稱篩選（存在/不存在教師）
  • • 教師專長篩選功能
  • • 預設顯示 48 筆教師記錄
  • • 依最近更新排序
  • • 日曆狀態顯示（已排課/未排課/請假/關閉時段）

課程詳情查看

  • • 課程完整資訊（課程 ID、教師、課程類型、封面圖）
  • • 學生清單（含頭像）與人數統計
  • • 扣除堂數、課程間隔、課程介紹
  • • 建立時間與課程時間顯示

登入與權限管理

  • • 必填欄位驗證（手機號碼、密碼）
  • • 手機號碼格式驗證
  • • 登入成功流程（機構選擇 → 首頁導向）
  • • 登入失敗處理（錯誤訊息提示）

測試執行的系統化方法

每個測試腳本都遵循統一的結構化格式，確保測試的可重複性和可追蹤性：

2. 測試步驟詳細：
• 操作流程逐步記錄

• 預期結果明確定義
3. 異常場景覆蓋：
• 必填欄位缺失
• 格式驗證錯誤
• 業務規則衝突
4. 跨平台驗證：
• 多螢幕尺寸（1440px / 320px）
• 多語言介面（中文 / 英文）
• 響應式設計檢查

技術整合：ClassIn API 的統籌管理

YesOnline 平台與 ClassIn 教學平台的深度整合，涉及9個 API 端點的協調管理。這不僅是技術整合的挑戰，更是產品經理跨系統統籌能力的考驗。

🔗 API 整合策略

課程管理 API

  • • 編輯課程資訊
  • • 課程下添加學生（單/多個）
  • • 課程下刪除學生（單/多個）

課節管理 API

  • • 修改課節資訊
  • • 課節下添加學生（單/多個）
  • • 課節下刪除學生（單/多個）
整合驗證：透過系統性測試確保 API 整合狀態，部分功能仍在開發驗證中，展現了產品管理中的風險識別和進度控制能力。
